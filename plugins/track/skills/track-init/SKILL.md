# Track Init

Bootstrap a project-tracking root at a user-specified path.

## Pre-flight

Look for an existing root **two ways**, in this order:

1. Walk up from cwd looking for `TRACKING.md`.
2. Check the user's global `CLAUDE.md` for a `## Project Tracking` section with a `Root:` path.

**Both checks are mandatory.** Walking up alone misses every root that isn't an ancestor of cwd — which is most of them, most of the time. That miss is how a second root gets built and a project's history gets split in two.

If a root exists either way, do NOT create another. Tell the user a root already exists at `<path>` and ask whether they want to add a project to it, reinitialize it, or deliberately create a separate root elsewhere.

## Process

### 1. Gather configuration (one question at a time)

1. **Where should the tracking root live?** (absolute path, e.g. `~/tracking`)
   - Mention both topologies so they can choose deliberately: one central root for all projects, or a root inside a specific repo. Both work; the marker file is what matters.
2. **Link style?** `relative` (default, portable) or `wikilink` (`[[…]]`, for Obsidian and similar tools)
3. **Naming language for generated files and directories?** (default `en`)

### 2. Create the structure

```
<root>/
├── TRACKING.md      ← marker + config (how /track finds this root)
├── README.md        ← project list, one line of current state each
└── <project>/       ← added in step 5, and by /track later
    ├── snapshot.md
    ├── history/
    └── refs/
```

### 3. Write `TRACKING.md`

```markdown
# Tracking Root

## Identity
- **Path:** <absolute path to root>
- **Created:** <YYYY-MM-DD>

## Config
- **link_style:** <relative | wikilink>
- **date_format:** YYYY-MM-DD
- **naming_language:** <en | zh | …>

## Layout
Each project is a directory under this root:
- `snapshot.md` — current truth. The only file read by default.
- `history/` — decisions only. One entry per fork in the road.
- `refs/` — external material, distilled on the way in.

## Conventions
- `snapshot.md` links only to decisions still in force.
- `history/` entries are never edited after the fact — a reversal is a new entry with `supersedes:`.
- Constraint IDs (`C1`, `C2`, …) are monotonic per project and never reused.
- This file is how `/track` locates the root. Do not move or delete it.
```

### 4. Write `README.md`

```markdown
# Tracked Projects

| Project | Status | Where things stand |
|---|---|---|
<!-- regenerated by /track from each project's snapshot -->
```

### 5. Scaffold the first project

An empty root is a dead end — end this run with something worth reading.

Ask for a project name, then create `<root>/<name>/` with `history/` and `refs/` (each holding a `.gitkeep`) and a `snapshot.md`.

Do not leave the snapshot as an empty skeleton. Interview the user for **Background** (what this project is and why) and **Where things stand** (current state + next step), and any **Constraints** they already know. Leave the rest as empty headings for `/track` to fill.

`snapshot.md`:

```markdown
---
project: <name>
updated: <YYYY-MM-DD>
status: active
---

## Where things stand

<one paragraph: current state + immediate next step>

## What's being tried

## Constraints

- [C1] <constraint, if the user named one>

## Key dates

## Options on the table

## Background

<what this project is and why it exists>
```

Section headings are fixed. `/track` reads and writes them by exact name — renaming or reordering them breaks it.

Then add the project's row to `README.md`.

### 6. Offer to register the root

**Do not skip this.** `TRACKING.md` marks the root, but it cannot be *found* from outside the root's own tree — `/track` walking up from an unrelated directory will never see it. Without a pointer, a later `/track` reports "no root found" and the user, following that advice, builds a second root. The project's history splits in two and nothing ever says so.

Ask:

> Want me to add this to `~/.claude/CLAUDE.md`?
>
> ```
> ## Project Tracking
> - Root: <absolute path to root>
> ```
>
> Without it, `/track` can only find this root when you're already inside it — and from anywhere else it may offer to create a second one.

- **Yes** → append the section (or add a `Root:` line if the section already exists).
- **No** → print the snippet and say where to paste it. Never write to a global file over an objection.

If the user has no global `CLAUDE.md`, offer to create one containing just this section.

### 7. Confirm

Tell the user:
- Root initialized at `<path>`, first project at `<path>/<name>`
- `TRACKING.md` is how `/track` identifies this root — do not move or delete it
- The `CLAUDE.md` pointer is how `/track` *finds* it from elsewhere — if the root ever moves, update both
- Run `/track <project>` after any session that moved the project forward
- `/track` writes only when asked; it never records anything on its own
